## Task tags

Projects have supported tags since v1, but tasks themselves have never had first-class support for
tags, until now. In v2.3, we're introducing [`tags`](/docs/config/project#tags-1) for tasks,
allowing you to label and organize tasks in a more flexible way. This opens up new possibilities for
filtering, grouping, and managing tasks based on their tags.

```yaml title="moon.yml"
tasks:
  lint:
    # ...
    tags: ['quality', 'ci']
```

To start, targets have been updated to support the `#` tag syntax in the task scope, allowing you to
do crazy things, such as:

- `:#tag` - Reference all tasks with the `tag` tag.
- `^:#tag` - Reference all tasks with the `tag` tag in upstream projects.
- `project:#tag` - Reference all tasks with the `tag` tag in a specific project.
- `#tag1:#tag2` - Reference all tasks with the `tag2` tag, in all projects with the `tag1` tag.

These targets can be utilized on the command line, enabling you to curate exactly which tasks to run
based on their tags.

```shell
$ moon run :#quality
```

Or they can be used in task dependencies, which will be expanded at runtime to a list of
fully-qualified targets.

```yaml title="moon.yml"
tasks:
  build:
    # ...
    deps: ['shared:#prereqs']
```

In addition to the new [`tags`](/docs/config/project#tags-1) setting, we've also added an
[`options.mergeTags`](/docs/config/project#mergeTags) setting to control how tags are merged during
task inheritance, and a [`taskTag`](/docs/concepts/query-lang#tasktag) field to MQL for querying
tasks by their tags.
